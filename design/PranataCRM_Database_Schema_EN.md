# PranataCRM — Database Schema Specification

> This document is a **database model generation instruction** for PranataCRM.
> Intended for junior developers or cheaper AI models that generate SQLAlchemy
> models + Alembic migrations. Follow ALL conventions in Section 1 without exception.
>
> Target stack: FastAPI + SQLAlchemy (async) + Alembic + PostgreSQL 15.
> Architecture: **modular monolith, NO foreign keys** (soft references).

---

## ⚠️ READ FIRST — Mandatory Rules (do not skip)

1. **NO FOREIGN KEYS.** Never write `ForeignKey(...)`, `REFERENCES`, or a
   `relationship()` that depends on an FK. All relations are stored only as plain
   UUID columns (soft references).
2. **Primary key = UUID v7**, generated in the application (Python), NOT by the
   database. Do not use `gen_random_uuid()` (that's v4) or the DB `uuidv7()`
   (not available in PostgreSQL 15).
3. **Every reference column MUST have a manual index.** Without an FK there is no
   automatic index. A missing index = slow queries.
4. **Tenant-scoped tables MUST have `tenant_id` + an RLS policy.**
5. Data integrity is enforced in the application layer (write guard, domain
   events, orphan sweeper) — NOT in the database. See Section 6.

When in doubt, default to: plain UUID column + index + no relational constraint.

## 1. General Conventions (apply to ALL tables)

### 1.1 Primary Key
- Column: `id`, type `UUID`, primary key.
- Value generated by the app via the `uuid7()` function (see Section 7).
- In SQLAlchemy: `default=uuid7`. Do NOT set a server_default.

### 1.2 Reference Columns (Soft Reference)
- Naming: `<entity>_id`, e.g. `tenant_id`, `contact_id`, `owner_id`.
- Type: `UUID`. May be `nullable=True` when the relation is optional.
- NO `ForeignKey`. Just a plain UUID column.
- MUST have an index (see 1.5).

### 1.3 Multi-tenancy
- Every tenant-owned table has a `tenant_id UUID NOT NULL` column.
- Such tables get an **RLS policy** (see 1.6).
- Exceptions (global/cross-tenant tables): `apps`, `permissions`, and `roles`
  rows where `is_system=true` (global templates). See per-table notes.

### 1.4 Timestamps
- `created_at TIMESTAMPTZ NOT NULL DEFAULT now()`.
- `updated_at TIMESTAMPTZ NOT NULL DEFAULT now()` (updated when the row changes).
- Use timezone-aware (TIMESTAMPTZ), not TIMESTAMP.

### 1.5 Indexes
- A manual index for EVERY reference column (`*_id`).
- Additional indexes for frequently filtered columns (e.g. `status`, `stage`,
  `type`, `email`).
- Unique indexes for columns that must be unique (e.g. `tenants.slug`).

### 1.6 Row-Level Security (RLS)
- Enable RLS on tenant-scoped tables.
- Policy: a row is visible only if `tenant_id = current_setting('app.current_tenant_id')::uuid`.
- The app sets `app.current_tenant_id` per request (out of scope for this doc).

### 1.7 Enums
- Store as `VARCHAR` + a `CHECK` constraint (not a PostgreSQL ENUM type, so it's
  easy to change). Example: `role VARCHAR(20) CHECK (role IN ('owner','admin','member'))`.

### 1.8 Flexible Data
- Use `JSONB` for dynamic fields (e.g. `metadata`, `custom_fields`, `settings`,
  `payload`).

### 1.9 Audit Fields (created_by / updated_by) — MANDATORY on tenant-scoped tables
- Every tenant-scoped table has `created_by` and `updated_by`, type `UUID`,
  `nullable=True`. These are **soft references → users.id** (NO FK, like any
  other reference column).
- `created_by` = the user who created the row. `updated_by` = the last user who
  changed the row. Lightweight "who-did-last" approach (not full history).
- Nullable because some rows are created by the system/automation (e.g. scheduled
  AI tasks, webhooks) with no acting user → set `NULL`.
- Implemented via `AuditMixin` (see Section 8). A model simply inherits the mixin
  and the columns are added automatically. Do NOT hand-write them per model when
  using the mixin.
- **Indexing `created_by`/`updated_by` is OPTIONAL** — only add it if you'll
  filter/report by these columns. This is an exception to rule 1.5 (audit fields
  are rarely used for filtering; avoid over-indexing).

Tables that use AuditMixin (created_by/updated_by):
`users`, `app_subscriptions`, `app_seats`, `roles`, `user_roles`, `companies`,
`contacts`, `deals`, `activities`, `ai_tasks`, `knowledge_base`, `report_snapshots`.

Tables that do NOT (system/global/mapping, never have a user actor):
`tenants` (created during signup bootstrap), `refresh_tokens` (system-managed),
`apps` & `permissions` (global), `role_permissions` (pure mapping),
`webhooks` (external events).

## 2. Module → Table Map

| Module | Tables |
|---|---|
| auth | `tenants`, `users`, `refresh_tokens` |
| licensing & authz | `apps`, `app_subscriptions`, `app_seats`, `permissions`, `roles`, `role_permissions`, `user_roles` |
| crm | `companies`, `contacts`, `deals`, `activities` |
| ai | `ai_tasks`, `knowledge_base` |
| billing | `webhooks` |
| reporting | `report_snapshots` |

Modular monolith note: tables owned by another module must NOT be accessed via
cross-module JOINs in application code. Cross-module access goes through the
contract/use-case (see ADR-007). At the DB level, cross-module relations remain
soft references only.

| Column | Type | Constraint | Notes |

| Column | Type | Constraint | Notes |
|---|---|---|---|

### 4.6 `role_permissions`
Many-to-many mapping of role ↔ permission.
RLS: follows role (accessed via app-layer join). `tenant_id` not required.

| Column | Type | Constraint | Notes |
|---|---|---|---|
| id | UUID | PK | uuid7 |
| role_id | UUID | NOT NULL | soft ref → roles.id |
| permission_id | UUID | NOT NULL | soft ref → permissions.id |
| created_at | TIMESTAMPTZ | NOT NULL DEFAULT now() | |

Indexes: `role_id`, `permission_id`; unique composite (`role_id`, `permission_id`).
Soft references: `role_id`, `permission_id`.

### 4.7 `user_roles`
Assignment of a role to a user, BOUND to a seat (app). See ADR-010.
RLS: YES (tenant_id).

| Column | Type | Constraint | Notes |
|---|---|---|---|
| id | UUID | PK | uuid7 |
| tenant_id | UUID | NOT NULL | soft ref → tenants.id |
| app_seat_id | UUID | NOT NULL | soft ref → app_seats.id (NOT directly to user) |
| role_id | UUID | NOT NULL | soft ref → roles.id |
| created_at | TIMESTAMPTZ | NOT NULL DEFAULT now() | |

Indexes: `tenant_id`, `app_seat_id`, `role_id`;
unique composite (`app_seat_id`, `role_id`).
Soft references: `tenant_id`, `app_seat_id`, `role_id`.
Important: a role is bound to an `app_seat`, not a global `user`. This ensures an
app role never "dangles" without a seat, and when a seat is revoked, the app role
disappears with it.

## 6. Maintaining Integrity Without FKs (3 Pillars)

Since there are no FKs, integrity is enforced in the application. When generating
services/use-cases (not models), remember these three pillars:

**Pillar 1 — Write Guard.** Before an insert/update that stores a reference,
validate that the reference exists (within the module: check the repository;
cross-module: check via the contract). Example: before creating a `deal`, ensure
`contact_id` exists.

**Pillar 2 — Delete Cascade via Domain Events.** When deleting a parent, publish a
domain event; the child-owning module cleans up its data (delete or set NULL).
Example when deleting a `contact`:
- `deals` with that `contact_id` → set `contact_id = NULL`.
- `activities` with that `contact_id` → delete.
- `ai_tasks` with that `contact_id` → set NULL or delete per policy.

**Pillar 3 — Orphan Sweeper.** A periodic job (Celery beat) scans for references
pointing to rows that no longer exist, then logs/cleans them. The last safety net.

List of relations to handle when a parent is deleted (for pillars 2 & 3):

| Parent deleted | Affected children | Suggested action |
|---|---|---|
| tenants | all tenant-scoped tables | delete (logical cascade) |
| users | app_seats, user_roles, refresh_tokens | delete; owner_id/user_id in crm → set NULL |
| companies | contacts.company_id, deals.company_id | set NULL |
| contacts | deals.contact_id (NULL), activities (delete), ai_tasks (NULL) | mixed |
| deals | activities.deal_id, ai_tasks.deal_id | set NULL |
| app_subscriptions | app_seats | delete |
| app_seats | user_roles | delete |
| roles | role_permissions, user_roles | block delete if in use (see ADR-010) |

## 7. Helper: UUID v7 Generator (Python)

Place this in `app/shared/types.py`. Used as `default=uuid7` on models.

```python
import os
import time
import uuid

def uuid7() -> uuid.UUID:
    """
    UUID v7 (time-ordered) generated in the application.
    Compatible with PostgreSQL 15 and below (no DB function needed).
    RFC 9562 layout: 48-bit ms timestamp | ver(7) | 12-bit rand_a |
    variant(0b10) | 62-bit rand_b.
    """
    unix_ms = int(time.time() * 1000) & ((1 << 48) - 1)
    rand = int.from_bytes(os.urandom(10), "big")  # 80 random bits
    value = unix_ms << 80
    value |= (0x7 << 76)                       # version 7
    value |= ((rand >> 68) & 0x0FFF) << 64     # 12-bit random_a
    value |= (0b10 << 62)                       # variant
    value |= (rand & ((1 << 62) - 1))          # 62-bit random_b
    return uuid.UUID(int=value)
```

Alternative: `pip install uuid6` then `from uuid6 import uuid7`.

## 8. SQLAlchemy Model Example (pattern to follow)

Follow this pattern for ALL models. Note: NO ForeignKey, plain UUID columns,
`default=uuid7`, indexes in `__table_args__`, timestamp mixin.

```python
# app/shared/base.py
import uuid
from datetime import datetime
from sqlalchemy import DateTime, func
from sqlalchemy.dialects.postgresql import UUID as PGUUID
from sqlalchemy.orm import DeclarativeBase, Mapped, mapped_column
from app.shared.types import uuid7

class Base(DeclarativeBase):
    pass

class UUIDMixin:
    id: Mapped[uuid.UUID] = mapped_column(
        PGUUID(as_uuid=True), primary_key=True, default=uuid7
    )

class TimestampMixin:
    created_at: Mapped[datetime] = mapped_column(
        DateTime(timezone=True), nullable=False, server_default=func.now()
    )
    updated_at: Mapped[datetime] = mapped_column(
        DateTime(timezone=True), nullable=False, server_default=func.now(),
        onupdate=func.now()
    )

class AuditMixin:
    """created_by / updated_by — soft reference to users.id (NO FK).
    Nullable: rows created by the system/automation may be NULL.
    Applied to ALL tenant-scoped models (see convention 1.9).
    Values are set in the service/use-case layer on insert/update."""
    created_by: Mapped[uuid.UUID | None] = mapped_column(PGUUID(as_uuid=True), nullable=True)
    updated_by: Mapped[uuid.UUID | None] = mapped_column(PGUUID(as_uuid=True), nullable=True)
```

```python
# app/modules/crm/models.py (example: contacts)
import uuid
from sqlalchemy import String, Index, CheckConstraint, Boolean
from sqlalchemy.dialects.postgresql import UUID as PGUUID, JSONB
from sqlalchemy.orm import Mapped, mapped_column
from app.shared.base import Base, UUIDMixin, TimestampMixin, AuditMixin

class Contact(Base, UUIDMixin, TimestampMixin, AuditMixin):
    # AuditMixin → created_by / updated_by added automatically (see convention 1.9)
    __tablename__ = "contacts"

    # Soft references — PLAIN UUID COLUMNS, NO ForeignKey
    tenant_id:  Mapped[uuid.UUID] = mapped_column(PGUUID(as_uuid=True), nullable=False)
    owner_id:   Mapped[uuid.UUID | None] = mapped_column(PGUUID(as_uuid=True), nullable=True)
    company_id: Mapped[uuid.UUID | None] = mapped_column(PGUUID(as_uuid=True), nullable=True)

    first_name: Mapped[str] = mapped_column(String(100), nullable=False)
    last_name:  Mapped[str | None] = mapped_column(String(100), nullable=True)
    email:      Mapped[str | None] = mapped_column(String(255), nullable=True)
    mobile_phone: Mapped[str | None] = mapped_column(String(50), nullable=True)
    job_title:  Mapped[str | None] = mapped_column(String(255), nullable=True)
    status:     Mapped[str] = mapped_column(String(20), nullable=False, default="lead")
    do_not_contact: Mapped[bool] = mapped_column(Boolean, nullable=False, default=False)
    tags:       Mapped[list] = mapped_column(JSONB, default=list)
    custom_fields: Mapped[dict] = mapped_column(JSONB, default=dict)
    # ... other columns per table definition 5.2 ...

    __table_args__ = (
        CheckConstraint(
            "status IN ('lead','qualified','customer','churned')",
            name="ck_contacts_status",
        ),
        # An index is MANDATORY for every reference column + filter column
        Index("ix_contacts_tenant_id", "tenant_id"),
        Index("ix_contacts_owner_id", "owner_id"),
        Index("ix_contacts_company_id", "company_id"),
        Index("ix_contacts_status", "status"),
        # created_by / updated_by: index is OPTIONAL (see convention 1.9)
    )
```

Note for the generator: do NOT add `relationship()` between models if it requires
an FK. If you need related data, fetch it via a separate query based on the UUID
column in the service layer.

## 9. RLS Notes (for migrations)

For each tenant-scoped table, add to the Alembic migration:

```sql
ALTER TABLE contacts ENABLE ROW LEVEL SECURITY;
CREATE POLICY tenant_isolation ON contacts
  USING (tenant_id = current_setting('app.current_tenant_id')::uuid);
```

For `roles` (which allows global system roles):

```sql
ALTER TABLE roles ENABLE ROW LEVEL SECURITY;
CREATE POLICY tenant_or_global ON roles
  USING (
    tenant_id = current_setting('app.current_tenant_id')::uuid
    OR tenant_id IS NULL
  );
```

Global tables (`apps`, `permissions`) do NOT need RLS.
